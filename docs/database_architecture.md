# QuantPrime Database & Storage Architecture (Level 4/5)

The QuantPrime application uses a **hybrid database architecture** that strictly separates high-volume, columnar analytical data (time-series) from low-volume, row-based transactional state (experiments, accounts, executions).

This separation is critical for achieving max performance in the execution pipeline (zero-copy operations) while maintaining relational integrity and sync capabilities for the UI dashboard.

---

## 1. Analytical Data Layer (DuckDB + Parquet)

The quantitative pipelines (ingestion, feature alignment, scoring, XGBoost ML, Risk APT) exclusively use **Apache Parquet files queried via DuckDB**. This is defined as the "Level 4 ECS" (Entity-Component-System) structure.

### Files & Paths
- **Directory**: `data/components/`
- **Mapping File**: `entity_map.parquet` (maps ticker strings to numeric `entity_id` to vastly improve query speed and memory footprint).
- **Core Parquet Files**:
  - `market_data.parquet` (daily price, volume, returns)
  - `fundamental.parquet` (quarterly SEC data with a strict 45-day lag guardrail)
  - `macro.parquet` (SPY, TNX, VIX, macro regime calculations)
  - `feature.parquet` (Standardized z-scores, factor betas, DCF gaps)
  - `action_intent.parquet` (Raw strategy weights proposed by ML models / Swarm)
  - `target_portfolio.parquet` (Final risk-adjusted weights after APT constraints)

### Why DuckDB + Parquet?
- **Zero-copy reads**: Polars/DuckDB can query Arrow-backed Parquet files directly from disk without serializing them into Python objects, enabling instant multi-gigabyte queries.
- **Columnar advantages**: Models often scan specific columns (e.g., `adj_close`) across millions of rows; columnar storage makes this orders of magnitude faster than SQLite.
- **Access Pattern**: Managed by `src/core/duckdb_store.py`.

---

## 2. Transactional App State Layer (Supabase Postgres)

Dashboard UI state, Alpha Lab AI experiments, Trader/Portfolio hierarchies, and Paper Executions are stored in a relational PostgreSQL database hosted on **Supabase**. This allows the frontend to have persistent, synced state.

### Core Tables
1. **`traders` / `trader_constraints`**: Account-level tracking and risk guardrails.
2. **`portfolios`**: Specific strategies assigned to a trader.
3. **`paper_executions`**: The ledger of simulated trades (ticks, quantities, prices).
4. **`alpha_lab_experiments`**: Persistent storage of LLM strategy generations, including hypothesis, raw code, token costs, performance metrics, and automated forensic audit verdicts (C1-C4).
5. **`editor_settings`**: Arbitrary JSON store for UI preferences (e.g., Swarm agent configurations).

### Access Pattern
- **ORM**: Handled safely via SQLAlchemy (`src/core/models.py`).
- **Endpoints**: Used extensively by FastAPI routers (`src/api/routers/`).
- **Security Guardrail**: Migrations and schemas are applied manually via SQL scripts rather than automated ORM pushes (`src/scripts/supabase_schema.sql`), enforcing human review of state changes.

---

## 3. Local Fallback Database (Offline / Dev Mode)

To ensure the trading platform functions in restricted cloud environments or locally offline, the transactional connection explicitly falls back to a **local SQLite database** if the Supabase Postgres endpoint times out or is unreachable.

### Implementation Details
- **Path**: `data/level3_trading.db`
- **Fallback Logic**: See `src/core/database.py`. The connection builder attempts a heartbeat `SELECT 1` against `DATABASE_URL`. If it fails, the connection transparently switches to the local SQLite file.
- **ORM Independence**: The SQLAlchemy models are strictly agnostic and work identically across both Postgres and SQLite without code changes.

---

## 4. Heavy Blobs & OOB Local Storage (Alpha Lab)

Some transactional state is too heavy for REST API boundaries (and too large for individual relational rows in Postgres). Specifically, the **dense discrete equity curves** generated by millions of backtests.

- **Storage**: Saved as local `.parquet` blobs in `data/alpha_lab/equity_curves/`.
- **Naming Convention**: `[experiment_id].parquet`.
- **Trade Ledger**: Individual backtest trade ledgers are saved to `data/alpha_lab/trades/trades_[experiment_id].parquet`.

These files are read on-the-fly when viewing a specific strategy in the Alpha Lab UI, preventing network saturation.
